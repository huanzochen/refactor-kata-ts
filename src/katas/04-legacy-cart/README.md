# Kata 4 — Legacy：先補特徵化測試，再重構 ⭐⭐⭐⭐

> 這題直接對應你最大的卡點：**面對看不懂、又沒測試的別人的 code，怎麼敢動？**
> 答案就是這套流程。

## 情境
`cart.ts` 的 `cartReducer` 是一段沒有測試、巢狀 if/else、手刻 for 迴圈的 legacy code。
你不完全理解它 —— **沒關係，重構本來就不要求你先 100% 理解。**

## 流程（重點在順序）

### 第一步：補特徵化測試（Characterization Test）
先不要改任何 production code。打開 `cart.test.ts`，把 `cartReducer`
**現在實際**的行為用測試釘住 —— 包含看起來很怪的行為（例如 `setQty` 到 0
不會刪除品項）。做法：
1. 給一個輸入，你猜一個輸出，跑測試。
2. 猜錯了？測試會告訴你真正的輸出 —— 把 assertion 改成真正的輸出。
3. 重複，直到覆蓋所有分支（add 新品項 / add 既有 / remove 存在 / remove 不存在 /
   setQty 存在 / setQty 不存在 / setQty 0 / setQty 負數）。

完成時你會有一張綠色的安全網，而且**你已經因此讀懂了這段 code**。
（注意順序：是「補測試」讓你讀懂，不是「先讀懂」才敢補測試。）

### 第二步：在安全網保護下重構
現在才動 `cart.ts`。目標把巢狀 if/else + for 迴圈換成清楚的結構：
- 用 `switch (action.type)` 取代巢狀 if/else。
- 用 `map` / `filter` 取代手刻 for 迴圈。
- 每一小步都跑測試，保持全綠。

### 第三步（可選）：修 bug —— 但要分開
如果你判定「setQty 到 0 應該刪除品項」是個 bug：**重構階段先別動它**
（那會讓安全網變紅、也混淆了「重構」與「改行為」）。等重構完、全綠之後，
再另外做一步：改行為 + 同步更新那條特徵化測試。

## 規則
- 第二步以後不要再改測試的「行為期望」（除非是第三步刻意改 bug）。
- 重構不改變任何對外行為。

## 完成的標準
- 測試涵蓋所有分支與邊界，且全綠。
- `cartReducer` 讀起來是一目了然的 `switch`。
- 你能說清楚哪個行為是「特意保留的現況」、哪個是「你之後修掉的 bug」。
