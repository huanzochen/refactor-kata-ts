# Refactoring Katas — React + TypeScript

給「會加功能、但不敢打掉重寫」的自己的重構練習場。
每一題都附一組**已經綠燈的測試**當安全網。你的工作只有一句話：

> **在測試從頭到尾保持綠燈的前提下，把 code 改乾淨。**

如果測試變紅，代表你不小心改變了行為 —— 那就不是重構了，退回去。

---

## 核心觀念（先讀這段）

重構不是「完全理解 → 打掉重寫」。恰恰相反：

**重構是一套讓你能安全修改「你沒有完全理解、也不敢動」的程式碼的技術。**

它靠兩根支柱：

1. **測試安全網** — 先讓行為被測試釘住，你才有底氣動手。看不懂沒關係，
   測試會在你改壞的當下告訴你。
2. **小步前進** — 每次只做一個有名字的機械化動作（抽函式、改名、搬移…），
   改完立刻跑測試。綠了才做下一步。永遠不要一次改一大坨。

你之所以覺得「只有自己剛寫完的 code 好重構」，是因為那時候你腦中就是安全網。
這些 kata 的目的，是讓你練成「**用測試當安全網，而不是用腦袋**」。

## 安裝 & 執行

```bash
npm install
npm test          # watch 模式，改 code 會自動重跑
npm run test:run  # 跑一次就結束（CI 用）
```

第一步：先跑 `npm test`，確認**全部綠燈**。這就是你的起點。

## 重構的節奏（每一題都照這個做）

```
1. 跑測試 → 全綠。這是你的安全網。
2. 讀 code，找出一個「味道」（重複、太長、命名爛、狀態糾纏…）。
3. 挑一個「有名字的手法」動它（見下方清單）。
4. 動一小步 → 存檔 → 看測試。
      綠 → 繼續下一步。
      紅 → 立刻 undo 回上一個綠燈點，想清楚再來。
5. 重複，直到你對這段 code 滿意。
6. 全程沒碰過測試檔案（除非題目要你補測試）。
```

**關鍵紀律：不要「順手」改行為。** 看到疑似 bug，先讓它留著、保持綠燈完成重構，
之後再開另一個 commit 修 bug。重構與改行為分開，是你能睡好覺的原因。

## 常用手法字典（Fowler 的簡版）

- **Extract Function / Hook** — 把一段邏輯抽成有名字的函式或 custom hook。
- **Rename** — 把爛名字改成說得出意圖的名字。
- **Introduce Parameter Object** — 一堆相關參數包成一個物件。
- **Replace Nested Conditional** — 巢狀 if 換成 early return 或 switch。
- **Consolidate State** — 多個會互相牽動的 `useState` 併成一個 `useReducer`，
  讓「不合法的狀態組合」根本無法表示。

  重構 UI 前先確認快照是綠的，改完再比對。
